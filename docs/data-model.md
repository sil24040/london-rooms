# Data Model Documentation

## Overview

LondonRooms stores application data in PostgreSQL. Supabase is used as the hosted PostgreSQL provider, while authentication is handled by the Express app through the `users` table, bcrypt password hashes, and JWT sessions.

Everything the application stores lives in Supabase PostgreSQL: users, room listings, payments, tenant-landlord messages, bookings, notifications, and reviews. The backend connects directly and runs SQL queries through the `pg` package.

The Supabase dashboard SQL Editor was used during development to create tables as features were added, inspect schema details when something was not behaving correctly, and manually fix development data, including reassigning demo rooms to real test accounts so enquiries and bookings showed up for the right landlord.

The current data model supports:

- Users and roles
- Room listings
- Saved rooms
- Enquiries
- Threaded messages
- Booking requests
- Tenant rental assignment
- Demo rent payment records
- Notifications
- Reviews

## Entity Relationship Summary

```text
users
  | 1-to-many rooms as landlord
  | 1-to-many enquiries as tenant or landlord
  | 1-to-many bookings as tenant or landlord
  | 1-to-many payments as tenant
  | 1-to-many notifications
  | 1-to-many reviews as tenant or landlord
  | optional rental_room_id -> rooms.id

rooms
  | many-to-one users as landlord
  | 1-to-many enquiries
  | 1-to-many bookings
  | 1-to-many payments
  | 1-to-many reviews
  | saved_by contains user ids

enquiries
  | many-to-one rooms
  | many-to-one users as tenant
  | many-to-one users as landlord
  | 1-to-many messages
```

## Tables

### `users`

Stores application accounts.

| Column | Purpose |
| --- | --- |
| `id` | Primary key |
| `name` | Display name |
| `email` | Unique login email |
| `password` | bcrypt password hash |
| `role` | `tenant` or `landlord` |
| `rental_room_id` | Optional room currently rented by a tenant |

### `rooms`

Stores room listings.

| Column | Purpose |
| --- | --- |
| `id` | Primary key generated by the app |
| `title` | Listing title |
| `description` | Listing description |
| `price` | Monthly rent |
| `area` | Area and postcode |
| `address` | Street or address |
| `type` | Room type |
| `bills_included` | Whether bills are included |
| `available_now` | Availability state |
| `landlord_id` | Owner user id |
| `landlord_name` | Denormalized landlord display name |
| `lat` / `lng` | Coordinates for map display |
| `image` | Optional uploaded image path |
| `saved_by` | Array of user ids that saved the room |
| `created_at` | Creation timestamp |

### `enquiries`

Stores tenant enquiries and simple landlord replies.

| Column | Purpose |
| --- | --- |
| `id` | Primary key |
| `room_id` | Related room |
| `room_title` | Denormalized room title |
| `room_area` | Denormalized room area |
| `room_price` | Denormalized room price |
| `tenant_id` | Tenant user id |
| `tenant_name` | Denormalized tenant name |
| `landlord_id` | Landlord user id |
| `landlord_name` | Denormalized landlord name |
| `message` | Initial enquiry text |
| `reply` | Simple landlord reply |
| `status` | Usually `pending` or `replied` |
| `created_at` | Creation timestamp |

### `messages`

Stores threaded messages attached to an enquiry.

| Column | Purpose |
| --- | --- |
| `id` | Primary key |
| `enquiry_id` | Related enquiry |
| `sender_id` | Sending user id |
| `sender_name` | Sender display name |
| `sender_role` | `tenant` or `landlord` |
| `body` | Message body |
| `read_by` | Array of user ids that have read the message |
| `created_at` | Creation timestamp |

### `bookings`

Stores booking requests.

| Column | Purpose |
| --- | --- |
| `id` | Primary key |
| `room_id` | Related room |
| `room_title` | Denormalized room title |
| `room_area` | Denormalized room area |
| `room_price` | Denormalized room price |
| `tenant_id` | Requesting tenant |
| `tenant_name` | Tenant display name |
| `landlord_id` | Listing landlord |
| `landlord_name` | Landlord display name |
| `message` | Optional booking request message |
| `status` | `pending`, `approved`, or `rejected` |
| `created_at` | Creation timestamp |
| `updated_at` | Last status update timestamp |

### `payments`

Stores rent tracker payments after Stripe test-mode confirmation.

| Column | Purpose |
| --- | --- |
| `id` | Primary key |
| `tenant_id` | Paying tenant |
| `room_id` | Rental room |
| `month` | Month in `YYYY-MM` format |
| `amount` | Room price at payment time |
| `card_last4` | Last four digits only |
| `paid_at` | Payment record timestamp |

### `notifications`

Stores in-app notifications.

| Column | Purpose |
| --- | --- |
| `id` | Primary key |
| `user_id` | Recipient user |
| `type` | Notification type, such as `enquiry_received` |
| `message` | Display message |
| `link_page` | Optional app page target |
| `read` | Read state |
| `created_at` | Creation timestamp |

### `reviews`

Stores tenant reviews for rooms.

| Column | Purpose |
| --- | --- |
| `id` | Primary key |
| `room_id` | Reviewed room |
| `room_title` | Denormalized room title |
| `tenant_id` | Reviewing tenant |
| `tenant_name` | Tenant display name |
| `landlord_id` | Listing landlord |
| `rating` | Integer from 1 to 5 |
| `comment` | Optional review comment |
| `created_at` | Creation timestamp |
| `updated_at` | Last update timestamp |

Recommended uniqueness:

- One review per `(room_id, tenant_id)`
- One payment per `(tenant_id, room_id, month)`
- One pending booking per `(room_id, tenant_id)`

## Validation and Business Rules

- Users must provide name, email, and an 8-character minimum password.
- Roles are tenant or landlord.
- Only landlords can create, update, or delete rooms.
- Only tenants can send enquiries, request bookings, set rentals, pay rent, and leave reviews.
- Tenants can only edit unreplied enquiries.
- Landlords can only reply to their own received enquiries.
- Landlords can only approve or reject bookings for their own rooms.
- Booking approval marks the room unavailable and assigns it as the tenant rental.
- Tenants can review only rooms they are eligible for through an approved booking or current rental assignment.
- Room image uploads must be image files and no larger than 4 MB.

## Denormalized Fields

Several tables store room and user names alongside foreign keys. This makes dashboard rendering simpler and preserves useful historical context if names or room details later change. The trade-off is that some updates must synchronize denormalized fields; for example, profile name changes update `rooms.landlord_name`.
